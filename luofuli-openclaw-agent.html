第一章

范式分水岭：OpenClaw 带来的认知革命

从"产品"到"划时代 Agent 框架"的重新定义

核心观点
OpenClaw 不是 Claude Code 加了个更好交互的 UI，而是一套「Agent 编排哲学」的开源具现。

罗福莉最初认为 OpenClaw 只是一个"偏运营导向的产品创新"，甚至排斥去用。真正的转折发生在她春节期间凌晨两点安装后，连续对话到天亮——连续三天，每天的惊喜层次不同：

第一天

感受到"产品有灵魂"——因为它做了很多精心的 Context 编排（如在每次对话前注入当前时间），营造出温度感。

第二天

发现它能替代一部分日常管理和工作思考，成为"数字分身"。

第三天

它真正帮助完成了研究任务——协助构建 user-agent 多轮交互的数据生成框架，而这原本被认为"不可能一两小时就做出来"。

洞见
框架设计本身是"智能涌现的放大器"。OpenClaw 之所以强大，是因为它的核心设计哲学是——用框架弥补模型短板，而不是依赖单一顶尖模型。

OpenClaw 的技术护城河：它具体好在哪里

这些单独看都有点"boring"，但组合在一起就形成了极高完成度：

任务覆盖编码为主含心跳任务、定时任务、日常场景
关键认知：Claude Code 所有设计都是为"写好代码"服务，OpenClaw 则是为"弥补模型短板、端到端完成任务"服务。后者的泛化性更强。

核心洞见
开源 Agent 框架真正的优势不是"免费"，而是能调动全人类智慧共同进化。

第二章

Agent 框架 ≠ 产品，它是"人与模型之间的中间层"

重新定义 Agent 框架

很多人把 OpenClaw 理解为"一个更好的产品"，这是错的。

❌ 错误理解
产品 = 人能直接感知交互的那一层（最薄的层）
✅ 正确理解
Agent 框架 = 人与模型之间的中间层，承担：对话 Context 编排、多模型调度与能力补位、持久化记忆管理、主动任务执行（心跳/定时/环境感知）、弥补模型行动上的缺陷
核心方程
最终任务完成率 = f（模型能力，Agent框架设计，用户 Context 沉淀）
对于中层模型，框架是"缺陷补位器"；对于顶尖模型，框架是"能力放大器"。

一个好的 Agent 框架应该具备什么

  1. 弥补行动上的缺陷：持久化记忆、多 channel 消息、定时主动任务、自我更新迭代
  2. 广泛的 Context 获取能力：模型获取不到的环境信息要主动补
  3. 完善的评估体系：框架自迭代的前提，当前大多数评估只是"防止致命错误"，远不够
  4. 容错性强的推理架构：Agent 场景下任务会中断、会超时，不能像预训练那样追求"零 Spike"

Skills 是"预训练无法覆盖的知识"的补充形式

类比
Skills ≈ 给博士生的导师手册 + 企业 SOP，这些是 GPT 训练数据里没有的。
  • 预训练数据覆盖的是互联网公开知识
  • 很多智能存在于企业内部、组织规范、个人经验中，互联网上根本没有
  • Skills 提供了一种交互机制，让人类主动贡献"私有知识"给 Agent
  • 这些知识以 Skill 的形式被 Agent 内化，大幅提升复杂场景任务完成率

第三章

训练范式的根本转变：从 Chat 到 Agent

为什么过去的"Agent"都不算真正的 Agent？

2025年初以前，大多数"Agent模型"的实质只是：

  • 换了一个更复杂的 System Prompt
  • 加了一点点环境反馈（如 bash 执行）
  • 没有工业级可用性：接上 Claude Code 或 OpenClaw 就露馅了

真正的 Agent 模型，需要能理解：

  • 整套框架的设计意图
  • 人与 AI 交互范式的转变（人不再"修改某行代码"，而是提更高层次的需求）
  • 跨 session 的持久记忆
  • 端到端完成长任务的能力

关键转变：长上下文 + Agent 场景的后训练

核心洞见
能做好 Coding 任务 = 能做好 Agent 的基础。Code 在三个范式（预训练、Reasoning、Agent）都是最优雅的训练路径。

为什么 Code 有如此强的泛化性：

  1. 预训练：代码是互联网上最长上下文依赖最密集的数据，训好 Code = 训好长上下文建模
  2. Reasoning：代码和数学都有可验证的 reward，是 RL 训练的完美场景
  3. Agent：软件开发本身就是极复杂的长链条任务，框架完善、环境反馈丰富
训练资源比例建议（当下）
研究卡 : 预训练卡 : 后训练卡 = 3 : 1 : 1

# 注：预训练 vs 后训练已从过去的 5:1 → 3:1 → 正在趋向 1:1

后训练的数据构造难题

后训练面临的核心挑战是如何构造真实的 Agent 环境数据：

  1. 需要还原用户与 Agent 交互的完整环境（不只是对话文本）
  2. 有了环境才能设计精准的 reward
  3. 长链条任务（1M context）的训练效率太低，实践上先用较短上下文激活，再外推
关键判断
谁能在「环境还原 × Reward 设计」这两个维度突破，谁就可能训出代差级模型。

第四章

Mimo VR 模型架构的核心决策

为什么选 Hybrid Attention，而非 MLA？

MLA 已经在"访存/计算"达到了完美的临界点，没有空间再上 MTP（Multi-Token Prediction）。

Hybrid Attention（滑动窗口 + 全局注意力混合）在计算上天然有大量富余，适合：

  • 上 MTP：充分利用富余算力，实现推理加速（Mimo Flash 能达到 100–150 TPS）
  • 更大 7:1 的稀疏比（Pro 比 Flash 更极致）：控制 KV Cache 增长，让大模型也能保持长文高效率
Flash 架构决策时间线
1. 预训练前：确定目标"长上下文高效率"，选 Hybrid Attention
2. 预训练中后期：发现计算有大量富余，加上 MTP
3. 验证：MTP 不引入幻觉（预测错误时不采纳），只加速不降质
核心原则：架构目标不应该太复杂，设定过多约束会导致后训练时条件全部失效。只 beat 一个点："长上下文效率"，其他随迭代演进。

Mimo VR 三模型矩阵的设计逻辑

不合并成一个模型，是基于成本、速度、效果的 trade-off：

🧠

大脑高层决策
👁️
Omni
全模态感知（视频/音频/图像/文本联合）

🔊

音频输出生成

编排原则：大部分任务用语言模型，在需要"感官能力"的环节调用 Omni，输出语音时调用 TTS。因为三个模型同一预训练数据，背景知识天然共通。

关于音频离散化的执念：团队将音频也离散化为类 token 的表示（多层 RVQ），统一到 LLM 范式。成功后发现：在少量风格化数据 SFT 后，模型能泛化到任意自然语言描述的风格，展示出极强的泛化性——这正是"简洁架构 × 大规模预训练"带来的涌现能力。

第六章

自学习与 AGI 的路径推断

Agent 自学习是怎么发生的

核心洞见
Agent 的自学习需要"模型"与"框架"同步进化，单方面进步不够。

自学习的发生路径：

  1. 框架中的静态信息（Prompt/System Prompt）在训练中应该随模型能力动态变化
  2. 框架本身的架构设计（动态信息）需要针对不同场景迭代
  3. 模型训练通过强化学习提升，反过来改变框架需要的上下文组织

当前缺失的两个层面：

  • 框架自身的自净化（Agent Framework 的自进化）
  • Agent 与人之间的共进化（人的经验更自然地被 Agent 吸收）

AGI 时间线的修正

2025年初判断
AGI 距今至少两年以上
访谈时（2026年初）修正
两年以内，即 2027–2028 年
修正原因：看到了一个此前认为"不可能"的事发生了——AI 能辅助做真正的模型研究，包括从研究方向、实验设计到代码实现的全链条。
AGI 的判断标准（她的定义）
不需要一个清晰的定义，当你发现"每个人的生活方式、工作方式已经被持续改变了很长时间"，那就到了。

路径预判：先替代工作方式，再替代生活方式；替代生活需要更多依赖机器人（硬件瓶颈）。

第七章

组织与团队的"反常识"选择

没有职级、没有分组的研究团队

100人
大模型团队总规模
20–40人
真正一代模型迭代主力
55%
博士比例（含在读）

管理方式：无职级、无固定分组（无"预训练组"/"后训练组"之分），且越来越偏向招大二大三本科生。

理由：

  • 职级和分组会扼杀创造力，尤其是扼杀跨领域流动
  • 做预训练的人去做后训练有天然优势：对数据多样性的直觉
  • 本科生"思想没有被污染"，对新范式的接纳能力更强

热爱驱动，不是管理驱动

"我觉得最有效的管理方式就是——激发热情，让大家围绕自己信仰的事情去做事。"

两个关键操作：

  1. 让人体验：OpenClaw 的推广不是讲道理，是让100人强制用，相互激发
  2. 筛人标准：不看历史经验，看"初始化的上限"（好奇心、热爱、技术基础），经验 1–3 个月就能补
这些能力都可以 create。我觉得最多一两个月，慢的话三四个月确实都可以被快速习得。所以环境反而比经验更重要。

训练 1T 模型的团队规模

几千张卡，但主力迭代人员约 20–40 人（含数据、算法、Infra）。

关键不是人多，而是：

  • 每个人追求极致、不容忍 Loss Spike
  • 问题出现时，小团队响应速度远快于大团队（跨团队协作效率极低）
  • 没有明确发布 deadline，发现问题就停下解决，哪怕停两周

第八章

最值得警惕的错误路径

"虚假 Agent" 陷阱

很多 2024–2025 年的模型在 SWE-bench、LiveCodeBench 等 benchmark 上表现很好，但：

  • 这些 benchmark 太 specific，只能测"修 bug"而非"做软件工程"
  • 模型只是换了更复杂的 System Prompt + 少量环境反馈
  • 接上 Claude Code / OpenClaw 就不可用
正确路径：在复杂 Agent 框架下，端到端完成高复杂度任务，以此为后训练目标。

MLA 在 Agent 时代的局限

MLA（Multi-head Latent Attention）在 Chat 时代是最优结构，但：

  • 不支持 MTP（MTP 会卡在计算瓶颈）
  • 推理速度相对慢
  • 在 post-train 周期拉长的 Agent 时代，预先固定推理场景的假设失效
但：这不代表 MLA 是"错误"，它是在受限算力下追求最强基座的正确选择；Hybrid 是在追求 Agent 效率的正确选择。两者目标不同。

第十章

一句话提炼每个核心洞见

1
OpenClaw 时刻：这不是"产品创新"，而是"Agent 编排哲学的开源具现"，让中等模型发挥顶级能力。

框架哲学：好的 Agent 框架的本质是"弥补行动缺陷"——把模型获取不到的上下文都喂给它。
